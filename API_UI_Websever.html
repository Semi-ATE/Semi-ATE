API zwischen UI und Webserver

Das User Interface (UI) hat die folgenden Aufgaben:

  1. Zustand des Testers mit allen seinen Sites darstellen

  2. Steuerung der sogenannten Masterapplikation um eine Testprogramm laden und entladen zu können

  3. Steuerung der Testapplikation. Einem gestarteten Testprogramm den Befehl zu geben sich auszuführen

Systemzustände

Um den Zustand des Testers darstellen zu können genügt es die Zustände der Masterapplikation zu betrachten. Es folgen die Zustände der der Masterapplikation:

  1. connecting: Tester verbindet sich mit allen Komponenten (MQTT Broker, Sites etc.)

  2. initialized: Der Tester befindet sich im initialen Zustand. D.h. es ist kein Testprogramm geladen. In diesem Zustand ist lediglich das Laden eines Los möglich. Dieser Zustand kann in den Zustand loading wechseln.

  3. loading: Der Tester ist aktuell damit beschäftigt ein Testprogramm zu laden. In diesem Zustand kann der Anwender keine Eingaben/Parametrierungen vornehmen.

  4. waitingforbintable: Der Tester ist aktuell damit beschäftigt die bin information zu empfangen. In diesem Zustand kann der Anwender keine Eingaben/Parametrierungen vornehmen.

  5. ready: Der Tester hat erfolgreich ein Los geladen. Der Anwender hat in diesem Zustand zwei Möglichkeiten. Er kann entweder das Los entladen oder den Befehl zum Teststart geben. Somit sind die Zustände unloading und testing möglich.

  6. testing: Der Tester ist mit der Ausführung eines DUT-Test beschäftigt. Dieser Zustand wird automatisch wieder verlassen, sobald der DUT-Test ausgeführt ist.

  7. finished: In diesem Zustand wird das Testprogramm beendet.

  8. unloading: Der Tester ist mit dem Entladen eines Testprogramms beschäftigt. In diesem Zustand sind keine Benutzereingaben möglich. Anschließend befindet sich der Tester im Zustand init.

  9. error: Prinzipiell kann von jedem Zustand aus der sogenannte Fehlerzustand error betreten werden.

Websockets

Der Datenaustausch zwischen der Webserverkomponente, im Folgenden als Backend (BE) bezeichnet und dem UI, im Folgendem als Frontend (FE) bezeichnet soll über Websockets stattfinden.

Datenformat

Das Datenformat zwischen FE und BE soll JSON sein. Jedes Datum besteht aus mindestens einem Key-Value-Pair type und einem optionalen Key-Value-Pair payload:

{
    "type" : <string>,
    "payload" : <number | string | {}> // optional
}

FE-Komponenten

Zunächst sollen die einzelnen FE-Komponenten betrachten werden.

Statuskomponente

Die Statuskomponente des FE ist für die Darstellung der nachfolgenden Informationen verantwortlich. Die hierfür notwendigen Daten werden vom BE als JSON geliefert:

  • Systeminformationen

    • systemName Name des Systems zB SCT 3000

    • environment zB. Final 1, Final 2, Final 3, Probe 1, Probe 2 etc.

    • handler Der DUT-Handler zB. Geringer, TEL P8, etc.

    • time System Zeit

    • log Log Informationen

    • program Der Name des geladenen Testprogramms

    • num_of_sites Anzahl der Sites

    • state ZB. connecting, initialized, loading, waitingforbintable, ready, testing, paused, unloading

Status

  • status

    {
    "type" : "status",
    "payload" :
    {
        "device_id": "<devic_id>",
        "systemTime": "<system time>",
        "sites": ["<sites>"],
        "state": "<master state>",
        "error_message": "<error_message>",
        "env": "<environment>",
        "lot_number": "<lot_number if loaded>"
    }
}

Usersettings

  • user settings

    {
    "type" : "usersettings",
    "payload" :
    {
        "testoptions":["<settings> z.B: {'name': 'stop_on_fail', 'active': false, 'value': -1}"],
        "loglevel": "<debug level>"
    }
}

Testresults

  • test results

    {
    "type" : "testresults",
    "payload" :
    [["<stdf records>"], ["<stdf records>"], "etc..."]
}

    Jede Liste [“<stdf records>”] entspricht einem Testprogramm-Lauf für ein Device/Part. Sie besteht somit aus mehreren STDF-Records und sollte aus genau einem PIR (Part Information Record), einem PRR (Part result Record) und mehreren Result Records (PTR, FTR, oder auch MPR) bestehen.

Controlkomponente

Die Controlkomponente wird zur Steuerung des Testers verwendet. Die Steuerung geschieht durch die Übertragung eines JSON vom FE an das BE.

  • Beispiele:

  • Lade Losnummer

    {
    "type" : "cmd",
    "command" : "load",
    "lot_number" : 123456.123
    "connectionid" : "<conneciton id>"
}

  • Starte Testausführung

    {
    "type" : "cmd",
    "command" : "start"
    "connectionid" : "<conneciton id>"
}

  • Starte Testausführung

    {
    "type" : "cmd",
    "command" : "unload",
    "connectionid" : "<conneciton id>"
}
